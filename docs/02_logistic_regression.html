Logistic Regression

Logistic Regression is a simple yet effective model, and makes for a good baseline. We will use it to introduce the metrics and techniques used to deal with high class imbalance, and to check whether using a more sophisticated model (gradient boosted tree) later improves performance.

Time Splits

When dealing with a time series, data needs to be split across time to respect causality and avoid information leakage. A single compromised payment method will generate several fraudulent transactions. If those transactions find themselves both in the test and training datasets, performance will increase artificially and the model will fail to generalise well.
As behaviour depends on the hour of the day, ideally we would want our time splits to feature representative parts of day/night. But with only two days of data we have to compromise: trying to fit a fair representation of night/day across all splits would break causality and inevitably cause time leakage.

Let us demonstrate the trap of information leakage by training the model on two different time splits:
-First one trains on 0-50% of the data and calibrates+tests on 70-100%
-Second one trains on 20-70% of the data and calibrates+tests on 70-100%

Though the training sample size is roughly the same, and test/calibration sets are identical, the 20-70 split performs substantially better as can be seen on the ROC and PR plots (see model evaluation section). This apparent performance boost is misleading: as data from the second day can be found both in training and testing sets, the model leverages data leakage to make better predictions, but this improvement will vanish on newer data.

This nicely demonstrates the necessity for a time buffer between training and testing data to avoid overly optimistic performances in production. In our case, with two days of data we can only go so far, so we will stick with split 1 (training only day 1) for the rest of this project.

Model Evaluation

For problems with high class imbalance such as this one, precision alone is not apropriate. A simple classifier that flags all transactions as genuine will have a precision of 100 - 0.17 = 99.83%, but would be completely useless. A useful fraud detection model must achieve both a high recall (the proportion of positives correctly classified, also known as the True Positive Rate) and a high precision.

Precision and recall, however, depend on the decision threshold. A model only provides probability estimates, and applies a default threshold of 0.5. For highly imbalanced data the threshold needs to be chosen carefully, as the low number of positives offers a lot of possible trade-offs. But we will concern ourselves with these trade-offs later. For now, let us simply vary that threshold across all possible values to obtain two threshold-independent metrics: the Precision Recall curve, and the Receiver Operating Characteristic (True Positive Rate against False Positive Rate), which are better suited for evaluating imbalanced models.

The performance of the logistic regression model will serve as a baseline for comparison with the more complex gradient-boosted decision tree.

In the meantime, let us note that the performances here look very good, and shall I say, -suspiciously- good. An AUC (Area Under the Curve) of .94 and of .70 for the ROC and PR curves respectively are what we might call "more than production ready" in another context, but here it is rather a testament to the limitations of the data set. Fraud patterns are probably really similar across the two-days, and the model learns to recognize those specific patterns instead of than what fraud might look like in general. As a concrete example, the model might learn to recognize specifics bursts of transactions as fraud and correctly classify a fraud case because it saw the exact same pattern during training. The model would be right in this limited context, but what was learned is not nearly as general as what the performance metric would lead us to believe.

Calibration

Motivation

In principle, a probabilistic classifier outputs estimated class probabilities. For example, among all transactions assigned a score of around 0.6, roughly 60% should indeed be fraudulent. A model that respects this relationship is said to be well-calibrated.

In practice, this property often breaks down when models are trained on highly imbalanced data, especially when class weighting is used. In such cases, the model tends to become overconfident: most fraudulent transactions receive very high scores, most genuine transactions receive very low scores, with comparatively fewer samples in between.

We can see this phenomenon on the recall/threshold curve. Even for thresholds as high as 0.995, the majority of positives are already detected.
This doesn't mean the model performs poorly, but a well-calibrated model makes dynamically choosing a threshold easier, for two reasons:

  1. The output probabilities are easily interpretable (around 0.6, the model is 60% confident).
  2. The changes of recall/precision with threshold are more gradual.

For these reasons, we map the output values of the model from raw scores to calibrated probabilities, typically using Platt scaling or an isotonic regression. Such a calibration changes the outputs of the model, but keeps the order of the predictions as the function we fit to the outputs is a monotonically increasing function. This means that performances that rely on the order of the predictions (PR and ROC curves) are not altered, as we can see on the following plots.

Limits

The calibration step tries its best to make the probability estimates accurate without tampering with the order. Because rankings are preserved, calibration can only correct probability estimates to a limited extent when the underlying model already separates positives and negatives extremely well. Those limits can be seen on the calibration curve we plotted for the test data. The curve is obtained by binning predictions by score and plotting, for each bin, the empirical fraction of positive samples against the predicted probability. For the uncalibrated model, only the last bin contains a significant portion of positives.
After calibration, the model attempts to align predicted probabilities with empirical frequencies, but two constraints remain:

  1. Data scarcity: the dataset contains fewer than 500 fraud cases in total, and only around 70 in the test set, leading to high variance in probability estimates.
  2. Preserved ordering: because ranking is unchanged, predictions above a threshold of roughly 0.3 are still overwhelmingly positive.

This is a nice reminder that calibration is a tool, not an end: a hypothetical perfect model that flags all frauds as 1 and all non-frauds as 0 would be impossible to calibrate for instance.

Threshold selection

As mentioned before, for data with high imbalance, sticking with the default 0.5 threshold is rarely appropriate. In fraud detection, achieving a high recall generally requires accepting a substantial number of false positives, as fraudulent transactions do not always stand out clearly from genuine ones. There are several strategies we can rely on to choose the threshold, most notably:
-Minimum recall approach: "We owe our clients to catch at least X% of the frauds". This can be limited, as it doesn't take into account the cost of a low precision.
-Cost function approach: "A false negative costs X, a false positive costs Y, what threshold minimizes the overall cost ?". This is a very reliable approach, granted you are able to estimate those costs correctly. This can be tricky, as the cost of mistakes includes a reputational cost on top of the monetary one. -Top K selection approach: if N agents can each go through M transactions during the day, then pick the K = M * N most suspicious transactions for verification.

Each of these strategies emphasizes a different operational constraint, and in real-world systems the final threshold is often chosen using a combination of them.

To keep things simple, we adopt a cost-based approach with the following cost function:

C=20FN+1FP

This reflects the assumption that missing a fraudulent transaction is approximately twenty times more costly than raising a false alert. While this ratio is admittedly arbitrary it could reasonably be set to 50 or even 100 the resulting optimal threshold is relatively stable.
As you can see on the figure below, there is a point below which the recall plateaus while the precision drops. Below is a plot of the cost function for different values of the relative cost between False Negatives and False Positives: 20, 50, 100. As you can see, the optimal threshold is at that tipping point in all three cases.

This robustness suggests that, despite the uncertainty in cost estimation, the chosen threshold captures a sensible trade-off between fraud detection effectiveness and operational burden.

We can now confidently say that a 0.007 threshold is optimal, and compute the corresponding performances on the test set: TPR = 81%, Precision = 40%

Again, those results are abnormally good because of the limited time span of the data, and it's exceptional quality.

Logistic Regression Conclusion

Logistic Regression provides a strong baseline and serves as a useful framework to introduce time-aware splitting, appropriate evaluation metrics for class imbalance, probability calibration, and cost-based threshold selection. While the resulting ROC and PR performances appear very high, they should be interpreted cautiously given the limited temporal span of the dataset and the risk of learning dataset-specific patterns. This baseline establishes a solid reference point for evaluating a higher-capacity model in the next section.
